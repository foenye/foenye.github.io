天平系统领域模块化落地说明 - 基线版本

1. 文档口径

这份文档描述基线落地方案：一个店主可以绑定多个 Shopee 店铺账号，一个店铺同一时刻只绑定一个运营，一个订单只归属这个店铺当前合作运营。advanced.md 描述多对多扩展版本，不在本文件展开。

设计前提：

  • 业务规则以原始业务需求为准。
  • 系统不假设 Shopee 没有提供的数据。订单、状态、退款、结算、调整，都以 Shopee 可同步到的数据为外部事实。
  • 文档使用“模块”描述边界。模块可以未来实现为微服务，也可以先做成单体内的领域包，但边界、数据所有权、流程所有权保持清晰。

系统核心一句话：

店主以邮箱作为天平唯一用户身份，可绑定多个 Shopee 店铺；每个店铺绑定唯一运营和一份有效合约；订单同步后先锁定店主预付款，运营发货，Shopee 拨款后天平按 escrow 实际结算金额分账；后续退款、补贴、扣款都追加独立流水，不改历史结算。

模块职责一句话：

  • 身份与授权模块：解决谁登录、谁绑定 Shopee、token 生命周期如何刷新和失效。
  • 店铺合作模块：解决一个 Shopee 店铺属于哪个店主、由哪个运营服务。
  • 合约模块：解决分账比例、有效期、保证金规则、运费责任。
  • 订单同步模块：把 Shopee 订单变成天平订单，并保存历史快照。
  • 托管锁款账户模块：用店主预付款锁住订单金额，决定运营是否可以发货。
  • 履约发货模块：把可发货订单交给运营处理，并同步 Shopee 状态。
  • 结算分账模块：以 Shopee escrow 实际到账为准，生成三方分账。
  • 账款调整模块：处理已结算后的退款、补贴、扣款、运费调整。
  • 账户流水模块：所有钱的变化都必须落流水，余额只是流水汇总结果。
  • 货币与汇率模块：统一定义店铺币种、账户币种、金额精度、汇率来源和换汇流水。
  • 提现与保证金模块：处理可提现余额、合作保证金、违约扣罚。
  • 通知与任务模块：把“需要人处理”的节点变成明确任务。
  • 排行与经营看板模块：按已入账佣金展示激励数据，并保护店铺隐私。

3. Shopee 能力边界

Shopee 只作为外部平台能力提供方，天平不能假设自己能控制 Shopee 内部结算、退款、拨款或物流规则。

本方案只依赖这些 Shopee 能力类型：

  • 授权能力：店主跳转 Shopee 授权，回调后获得 code 和 shop_id，再换取 access_token 和 refresh_token。
  • 店铺能力：通过 shop_id 识别 Shopee 店铺，并把 Shopee 店铺绑定到天平邮箱用户。
  • 订单能力：同步 Shopee 订单号、订单状态、订单明细、包裹或物流相关信息。
  • 履约能力：读取发货、揽收、配送、签收、取消、退货等状态，必要时让运营按 Shopee 规则完成发货动作。
  • 结算能力：通过 escrow 明细获取最终结算金额和费用拆分。
  • 售后能力：读取退货退款状态和金额变化。
  • 调整能力：读取结算后补贴、扣款、退款、运费调整等账款变化。

接入实现按 Shopee 当前开放平台版本映射到具体接口。按现有材料，必须覆盖：

能力 典型接口/数据
授权换 token get_access_token、授权回调 code、shop_id
店铺识别 以授权回调和 token 关联的 shop_id 为准，不使用 main_account 作为当前落地主体
订单详情 get_order_detail、order_sn、订单状态、订单行
包裹和物流 get_shipment_list、get_package_detail、package_number、tracking_number
拆包能力 split_order，仅在 Shopee 规则允许时使用
结算明细 get_escrow_detail、escrow_amount、费用拆分字段
退货退款 get_return_detail 或同等售后查询能力
账款调整 escrow 明细中的调整字段和 Shopee 对账数据

系统边界必须说清楚：

  • 天平不能直接决定 Shopee 给店主虾皮钱包拨多少钱，只能按 Shopee 返回的 escrow 和调整数据做内部账。
  • 天平不能绕过 Shopee 规则强行改变订单生命周期，只能同步状态并在天平内部派生任务。
  • 天平的“托管锁款账户”是内部资金保障机制，不是 Shopee 平台托管账户。
  • 天平的“已结算”是内部账户流水已完成，不等同于 Shopee 订单状态字段。
  • 天平的“运营责任”是内部合作责任，不是 Shopee 官方订单归属。
  • 当前基线版本不做 main_account 体系。即使店主使用不同 Shopee 店主账号授权，只要授权店铺 shop_id 在天平归属到同一个邮箱用户，就进入同一个店主名下。

4. 模块一：身份与 Shopee 授权模块

4.1 模块使用者与系统行为

这个模块只负责登录、身份管理、角色管理、店主 Shopee 授权绑定、店主 Shopee token 保持有效、管理员用户管理。

使用者：

  • Management：管理员。管理账号、角色、账号状态、Shopee 授权异常。
  • Shopkeeper：店主。登录天平账号，绑定 Shopee 店铺 或 Shopee 店主账号授权流程后直接登录。
  • Operator：运营供货发货商登录。
  • SystemJob：系统任务。周期刷新 token、处理授权到期。

系统行为清单：

  • 全部用户支持使用用户名密码或验证码登录。
  • 支持直接 Shopee 授权店主账号登录，但首次必须补全邮箱。
  • 登录成功后返回 Token 或 Cookie。
  • 个人信息接口返回用户基础信息和角色列表。
  • 管理员新建、修改、删除、禁用用户。
  • 管理员给用户分配或移除角色。
  • 店主账号新增 Shopee 店铺授权(增加门店)。
  • 周期任务保持正常合作关系内的店主 Shopee token 有效。

管理员、店主、运营都走同一个登录流程。登录接口只校验身份并返回会话。

    B -->|是| D{账号是否禁用}
    D -->|是| E[拒绝登录: 账号禁用]

个人信息接口：

flowchart TD
    A[请求个人信息] --> B{Token/Cookie 是否有效}

4.4 管理员管理用户流程

管理员管理用户只改变天平账号和角色，不直接创建 Shopee 店铺。

补充：

  • 一个用户可以有多个角色。
  • 没有角色的用户可以登录，但个人信息接口返回空角色列表。
  • 禁用用户不能登录，已有会话在下次请求时会被已撤销拦截退出登录。
  • 删除有订单的用户做软删除或禁用，避免破坏历史订单、授权、流水审计。

4.5 已有店主账号新增 Shopee 店铺授权流程

已被分配 Shopkeeper 角色的账号，如果还没有店铺，或想继续新增店铺，走 Shopee 授权绑定流程。授权成功后系统直接从 Shopee 获取店铺信息，不能手动创建店铺。

sequenceDiagram
    participant 店主
    participant 天平
    participant Shopee

    店主->>天平: 请求绑定 Shopee 店铺
    天平->>天平: 校验当前用户有 Shopkeeper 角色
    天平->>天平: 生成 state 并绑定当前 user_id
    天平->>Shopee: 跳转授权
    Shopee-->>天平: 回调 code + shop_id
    天平->>天平: 校验 state
    天平->>Shopee: code 换 access_token / refresh_token
    天平->>Shopee: 使用 token 获取店铺信息
    天平->>天平: 校验 shop_id 是否已绑定其他用户
    天平->>天平: 自动落库/更新 OpenidConnector 和 Shop
    天平-->>店主: 返回绑定结果

补充：

  • 授权回调先校验 state，防止店铺串绑到错误邮箱用户。
  • 授权成功后必须从 Shopee 获取店铺资料并自动落库/更新，用户不能手动创建 Shopee 店铺。
  • 一个 shop_id 同一时间只能绑定一个有效店主邮箱。
  • 如果 shop_id 已绑定当前邮箱用户，本次授权更新 token 和店铺资料。
  • 如果 shop_id 已绑定其他邮箱用户，本次授权拒绝并转管理员处理。
  • token 只允许服务端授权适配层读取和使用，不暴露给前端、普通后台页面、导出和通知。
  • 日志、审计、异常堆栈、任务 payload 禁止输出 token 原文。

4.6 直接 Shopee 授权登录的首次补邮箱流程

店主也可以从 Shopee 授权入口直接进入。第一次进入时，系统只有 Shopee shop_id，还没有天平邮箱用户，因此必须补全邮箱后才能建立天平身份。

flowchart TD
    A[Shopee 授权回调 code + shop_id] --> B[校验 state]
    B --> C[换取 token]
    C --> D[获取 Shopee 店铺信息]
    D --> E{shop_id 是否已绑定用户}
    E -->|已绑定| F[更新授权并创建登录会话]
    E -->|未绑定| G[生成待补邮箱记录]
    G --> H[用户提交邮箱]
    H --> I{邮箱是否已存在}
    I -->|不存在| J[创建 User 并分配 Shopkeeper 角色]
    I -->|已存在| K[验证码校验后绑定到已有 User]
    J --> L[绑定 shop_id 到 User]
    K --> L
    L --> M[创建登录会话]

补充：

  • 邮箱是天平用户身份，必须唯一。
  • 邮箱已存在时，必须完成验证码校验后才能绑定到该用户。
  • 邮箱不存在时，创建新用户并默认分配 Shopkeeper 角色。
  • 补邮箱完成前，不创建可用登录会话，不进入后续业务流程。

4.7 店主重新授权流程

Shopee 提前 7 天通知授权到期(Shopee会推送消息)，或 refresh token 失效时，系统要求店主重新授权。重新授权只更新原 shop_id 的授权连接，不创建新店铺，不改变历史订单。

flowchart TD
    A[系统生成重新授权任务] --> B[店主发起重新授权]
    B --> C[Shopee 回调 code + shop_id]
    C --> D{shop_id 是否属于当前店主}
    D -->|否| E[拒绝并记录异常]
    D -->|是| F[换取新 token]
    F --> G[更新 OpenidConnector]
    G --> H[恢复该店铺新增同步相关能力]

4.8 店主 Shopee token 周期刷新任务

token 刷新任务只处理“正常合作关系内”的店铺授权，避免给已经停用、解绑、无合作关系的店铺持续刷新凭证。

纳入刷新范围的条件：

  • OpenidConnector.status = ACTIVE。
  • Shop.status = ACTIVE。
  • 店铺存在当前有效 ShopCooperation。
  • 合作关系绑定的合约当前有效。
  • Shopee 授权未超过 365 天有效期。

不纳入刷新范围的情况：

  • 店铺已解绑。
  • 店铺合作关系已终止，且没有未完结订单、售后、调整。
  • 授权已经失效并要求店主重新授权。
  • 店铺被管理员停用。
flowchart TD
    A[周期刷新任务启动] --> B[扫描正常合作关系内的有效店铺授权]
    B --> C{access_token 是否将在刷新窗口内到期}
    C -->|否| D{Shopee 授权是否 7 天内到期}
    C -->|是| E[使用 refresh_token 刷新 access_token]
    E --> F{刷新是否成功}
    F -->|成功| G[更新 access_token 和过期时间]
    F -->|失败| H[记录失败次数和失败原因]
    H --> I{refresh_token 是否仍有效}
    I -->|是| J[下次任务继续重试]
    I -->|否| K[标记 REAUTH_REQUIRED]
    D -->|是| L[通知店主重新授权]
    D -->|否| M[结束]
    G --> D
    J --> D
    K --> N[标记该店铺授权不可用]
    L --> M

补充：

  • access_token 有效期 4 小时；刷新任务只刷新即将到期或已到期但 refresh_token 仍有效的 access token。
  • refresh_token 有效期 30 天；refresh token 过期后不能靠系统自动恢复，必须通知店主重新授权。
  • 刷新失败不删除旧 token，系统记录失败原因，下一轮继续处理。
  • 连续刷新失败但 refresh token 未过期时，后续 Shopee API 调用按返回结果判断；调用 Shopee 返回 token 失效时，立即进入重试刷新。
  • 标记 REAUTH_REQUIRED 后，该模块只输出授权不可用状态；订单、履约、结算等模块根据该状态暂停各自的 Shopee 调用。
  • 店主重新授权成功后，复用原 shop_id、店铺档案、合作关系、合约和历史订单，只更新授权连接。

5. 模块二：店铺合作模块

5.1 模块使用者与系统行为

这个模块管理店铺、店主、运营之间的合作关系。它回答一个问题：这个店铺当前由哪个运营服务，这段合作从什么时候开始，到什么时候结束，绑定哪一份签署合约。

基线版本的合作关系规则：

  • 一个店主可以拥有多个店铺。
  • 一个运营可以服务多个店铺。
  • 一个店铺同一时刻只能有一个店主、一个运营、一份当前有效签署合约。
  • 合作关系必须有开始时间、结束时间、状态、签署合约版本。
  • 合作关系可以更换运营，但不能覆盖历史合作关系。

使用者：

  • 店主：发起合作、选择运营、查看合作状态、申请终止合作。
  • 运营：确认或拒绝合作邀请。
  • 管理员：审核合作异常、强制终止合作、处理运营更换。

系统行为清单：

  • 店主为已授权店铺发起运营合作。
  • 运营接受或拒绝合作邀请。
  • 合作接受后等待合约签署版本。
  • 合约签署后启用店铺合作关系。
  • 店主或管理员终止合作并保留历史记录。
  • 店主或管理员更换运营时，先终止旧合作，再创建新合作。

6. 模块三：合约模块

6.1 模块使用者与系统行为

合约决定钱如何分，也决定合作双方在违约、保证金、退货运费、提现等场景下的责任。原始业务需求已明确分账比例以店铺合作协议为准，不同店铺可以不同。

使用者：

  • Management：管理员。维护合约模板、录入敏感字段、核对 AI 风险提示、确认待签署版本。
  • Shopkeeper：店主。查看合约内容并签署。
  • Operator：运营。查看合约内容并签署。

系统行为清单：

  • 管理员维护合约模板和敏感字段规则。
  • 系统按模板生成合约草稿。
  • 大模型辅助审阅草稿并输出风险提示。
  • 管理员核对敏感数字、比例、保证金、运费责任、有效期。
  • 管理员确认后保存待签署版本。
  • 店主和运营签名或上传签字盖章文件。
  • 系统生成不可变签署版本和内容哈希。
  • 店铺合作关系绑定签署版本后才允许启用。
  • 订单创建时复制签署合约快照。

系统里的“合约快照”不是普通配置快照，而是具备法律效力的签署版本：

  • 合约必须从管理员维护的合约模板生成。
  • 店主、运营、店铺、比例、保证金、运费责任、有效期等敏感字段必须来自模板字段或管理员录入字段。
  • 大模型可以参与提交前审阅，检查条款缺失、比例不等于 100%、保证金低于业务门槛、运费责任与业务要求冲突、敏感数字异常等问题。
  • 大模型只输出风险提示和待核对项，不自动判定合约有效，不自动保存，不替代管理员审核。
  • 最终必须由管理员核对后才能保存为待签署版本。
  • 店主和运营完成签名，或平台上传双方签字/盖章文件后，系统才生成不可变的签署版本。
  • 订单、结算、退款、账款调整读取的是签署版本的合约快照，不读取草稿，也不读取未签署版本。

一份合约至少需要表达：

  • 店主佣金比例。
  • 平台佣金比例。
  • 运营回款比例。
  • 店主保证金要求，最低 15000 台币。
  • 运营保证金字段保留，当前最低门槛为 0。
  • 退货相关运费由运营统一承担。
  • 生效时间和失效时间。
  • 模板编号和模板版本。
  • 管理员审核人、审核时间、审核意见。
  • 店主签署信息、运营签署信息、盖章/签名文件。
  • 签署版本哈希，用于证明签署后内容未被修改。

结算时，店主佣金、平台佣金、运营回款按合约比例和订单币种精度计算。尾差归属必须有固定规则，默认由运营回款承接：

店主佣金 = 按币种精度处理(可分账金额 * 店主比例)
平台佣金 = 按币种精度处理(可分账金额 * 平台比例)
运营回款 = 可分账金额 - 店主佣金 - 平台佣金 - 运营承担费用 - 尾差调整

当前业务需求中平台佣金“暂不收取”，但模型仍保留 platformCommissionRate，便于以后开启。

7. 模块四：订单同步模块

7.1 模块使用者与系统行为

订单同步模块负责把 Shopee 订单变成天平订单。它只认 Shopee 订单事实，不自己发明订单状态。同步入口包括 Shopee 消息推送和系统周期同步；推送只负责触发拉取详情，真正入库以 Shopee 详情接口返回为准。

使用者：

  • SystemJob：系统任务。消费 Shopee 推送、主动拉取 Shopee 订单、执行周期同步、重试失败同步任务。
  • Shopkeeper：店主。查看自己店铺订单和订单历史。
  • Operator：运营。查看分配给自己的可履约订单。

系统行为清单：

  • 接收 Shopee 订单、物流、售后、结算相关推送。
  • 推送先写入事件收件箱，再触发详情拉取。
  • 周期任务主动拉取 Shopee 订单、物流、售后、结算相关详情。
  • 调用 Shopee 详情接口读取完整订单事实。
  • 创建或更新本地订单、订单行和订单历史快照。
  • 订单创建时复制店铺合作和合约快照。
  • 待发货订单发送给托管锁款账户模块。
  • 已完成订单发送给结算分账模块检查 escrow。
  • 售后或账款变化发送给账款调整模块。
  • 周期同步任务持续推进重点状态订单。

核心规则：

  • Shopee 订单号 order_sn 是外部唯一键。
  • 天平订单号可使用 X + order_sn。
  • 订单属于一个店铺。
  • 订单创建时锁定店主、运营、合约快照。
  • 每次同步都保存订单历史快照，便于追溯状态变化。
  • Shopee 推送不能直接改变订单，只能触发详情拉取；所有订单事实以 Shopee 详情接口返回为准。
  • 周期同步任务只扫描需要进入业务线或持续推进的订单：待发货、已发货、揽收失败、待签收、已完成待结算、售后中、已结算后可能有账款调整的订单。

OrderStatus 只保存 Shopee 同步来的订单状态。天平托管锁款账户是否完成，不能写进 OrderStatus，要看托管记录。

  • UNPAID：Shopee 订单待支付。
  • READY_TO_SHIP：Shopee 订单待发货。能不能通知运营，还要看托管锁款账户是否已锁款成功。
  • PROCESSED：运营已操作发货。
  • RETRY_SHIP：三方物流揽收失败，需要重新发货。
  • SHIPPED：物流已揽收。
  • TO_CONFIRM_RECEIVE：等待买家确认签收。
  • IN_CANCEL：买家发起取消申请。
  • CANCELLED：订单已取消。
  • TO_RETURN：买家发起退货退款。
  • COMPLETED：订单已完成，可进入拨款检查和结算。

8. 模块五：托管锁款账户模块

8.1 模块使用者与系统行为

托管锁款账户是业务核心。运营发货前，系统必须先从店主“我的预付款”中锁定订单托管金额，转入临时托管账户。这样运营发货后，结算时能收到回款。

使用者：

  • SystemJob：系统任务。处理待托管订单、充值后重试托管。
  • Shopkeeper：店主。收到预付款不足提醒并充值。
  • Operator：运营。只接收锁款成功后的发货任务。

系统行为清单：

  • 接收 Shopee 待发货订单。
  • 读取订单托管金额和店主预付款余额。
  • 余额足够时扣减预付款并转入临时托管账户。
  • 托管成功后记录托管状态为 LOCKED，再通知运营发货。
  • 余额不足时记录托管状态为 FAILED，订单状态仍保持 Shopee 同步状态。
  • 通知店主充值。
  • 充值后自动重试待托管订单。
  • 预付款不足超过 24 小时后通知平台介入。

托管金额口径：

天平托管金额以 Shopee escrow_amount 或可确定的最终结算口径为准，不直接用包含手续费的订单总价。

如果 Shopee 订单进入待发货，但店主预付款不足：

  • 订单状态仍按 Shopee 同步结果保存。
  • 托管锁款账户生成一条失败记录。
  • 通知店主充值。
  • 店主充值任意金额后，自动触发一次托管检查。
  • 超过 24 小时仍未充值，通知平台介入。
  • 未终止合作前，每 6 小时提醒一次。

9. 模块六：履约发货模块

9.1 模块使用者与系统行为

履约发货模块处理运营发货和物流状态。订单锁款成功后，运营看到发货任务；订单发出并完成后，系统再进入结算检查。

使用者：

  • Operator：运营。查看待发货任务、采购备货、按 Shopee 规则发货。
  • SystemJob：系统任务。同步包裹、物流、揽收、签收和售后状态。
  • Shopkeeper：店主。查看订单履约进度。

系统行为清单：

  • 锁款成功后生成运营发货任务。
  • 运营只能看到分配给自己的发货任务。
  • 运营按 Shopee 规则处理发货。
  • 系统同步物流揽收、发货、签收、取消、退货状态。
  • 揽收失败时生成重试任务。
  • 订单取消或进入售后时关闭未完成发货任务。
  • 订单完成后触发结算检查。

运营只能看到已经锁款成功的订单。未锁款订单不能让运营发货，否则业务中“保障运营发货后能收到货款”的规则就失效。

发货后要持续同步 Shopee 状态：

  • 运营已处理：PROCESSED
  • 物流揽收失败：RETRY_SHIP
  • 已揽收：SHIPPED
  • 等待买家确认：TO_CONFIRM_RECEIVE
  • 买家发起取消：IN_CANCEL
  • 买家发起退货：TO_RETURN

10. 模块七：结算分账模块

10.1 模块使用者与系统行为

结算以 Shopee 拨款到店主虾皮钱包为触发依据。原始业务需求明确：“虾皮结算到店主的虾皮钱包，天平系统以虾皮钱包到账为准触发内部结算。”

使用者：

  • SystemJob：系统任务。按事件或周期读取 escrow 明细。
  • Shopkeeper：店主。查看佣金和结算明细。
  • Operator：运营。查看运营回款。

系统行为清单：

  • 接收 Shopee 完成或结算相关事件。
  • 读取并保存 get_escrow_detail 原始明细。
  • 使用订单签署合约快照计算店主佣金、平台佣金、运营回款。
  • 释放订单托管金额。
  • 向账户流水模块发起三方记账。
  • 生成订单结算单并标记天平已结算。
  • escrow 暂未生成时由周期任务继续检查。
  • 结算完成后禁止覆盖重算，只允许追加账款调整。

结算数据来源：

  • 使用 Shopee get_escrow_detail 获取结算明细。
  • 以 escrow_amount 作为最终可对账金额。
  • 商品金额、正向运费、逆向运费、Shopee 佣金、服务费、交易费等明细用于透明展示和费用归属。

结算动作：

  • 托管账户释放订单锁定金额。
  • 店主佣金入店主佣金账户。
  • 平台佣金入系统佣金账户，owner_user_id = 0，当前金额可按 0 处理。
  • 运营回款入运营的佣金/回款账户。
  • 如果退货运费、逆向运费或需运营承担费用存在，则从运营回款中扣减。

11. 模块八：退款、退货、账款调整模块

11.1 模块使用者与系统行为

业务的核心原则是：

虾皮收款天平就收款，虾皮退款天平就退款；所有已结算后的变化追加流水，不修改原结算记录。

使用者：

  • SystemJob：系统任务。同步 Shopee 退款、退货、补贴、扣款、运费调整。
  • Management：管理员。处理无法自动判定责任的调整。
  • Shopkeeper：店主。查看退款、补贴、扣罚和保证金影响。
  • Operator：运营。查看回款扣减、退货运费和补贴影响。

系统行为清单：

  • 识别 Shopee 账款变化类型。
  • 判断订单是否已经完成天平结算。
  • 结算前变化更新待结算金额口径。
  • 结算后变化创建独立账款调整单。
  • 按原签署合约和结算比例快照生成回冲或补发明细。
  • 逆向运费默认扣运营回款。
  • 罚款按责任扣保证金或回款。
  • 无法自动判定责任时创建人工处理任务。

场景处理：

  • 发货前取消：如果已托管，释放托管金额回店主预付款；如果未托管，无资金动作。
  • 结算前部分退款：Shopee 实际拨付已扣减，天平直接按扣减后的 escrow_amount 分账。
  • 已结算后部分退款：生成负向调整流水，按原分账比例三方退回。
  • 活动补贴：生成正向调整流水，按原分账比例分配，除非业务规则明确归运营。
  • 逆向运费：由运营统一承担，影响运营回款。
  • 罚款：判断责任方，店主责任扣店主保证金，运营责任扣运营回款或运营保证金。

12. 模块九：账户与流水模块

12.1 模块使用者与系统行为

业务账户共 8 类：

使用者：

  • SystemJob：系统任务。执行托管、结算、调整、提现、扣罚等记账请求。
  • Shopkeeper：店主。查看预付款、佣金、保证金账户和流水。
  • Operator：运营。查看回款、保证金账户和流水。
  • Management：管理员。对账、处理异常流水、查看审计轨迹。

系统行为清单：

  • 初始化平台、店主、运营相关账户。

  • 接收业务模块的记账请求。

  • 校验账户状态、币种、余额规则和幂等键。

  • 锁定账户并写入不可删除流水。

  • 更新余额和冻结余额。

  • 保存变动前后余额。

  • 支持按订单、结算、调整、提现、扣罚业务单号反查流水。

  • 跨币种入账时关联汇率和换汇流水。

  • 平台账户：系统佣金账户、罚补账户、临时托管账户。

  • 店主账户：预付款账户、店主佣金账户、店主保证金账户。

  • 运营账户：运营佣金/回款账户、运营保证金账户。

账户使用统一账户表承载不同账户类型：

  • 店主佣金账户和运营佣金/回款账户使用同一种 COMMISSION 账户类型，通过账户 owner 区分归属。
  • 系统佣金账户也使用 COMMISSION 账户类型，owner_user_id = 0 表示系统账户。
  • 预付款、托管、保证金、罚补账户仍通过 account_type 区分。
  • 不再单独拆 CommissionAccount、OperatorReceivableAccount、PlatformCommissionAccount 多套账户表。

账户必须带币种。店主在台湾站经营时，预付款、托管、佣金、回款、保证金通常使用该店铺对应结算币种，例如 TWD；如果未来接入不同站点或跨币种提现，不能把金额简单相加，必须通过货币与汇率模块换算。

13. 模块十：货币与汇率模块

13.1 模块使用者与系统行为

货币与汇率模块负责回答四个问题：

使用者：

  • Management：管理员。维护汇率来源、汇率时间和币种规则。

系统行为清单：

  • 维护币种代码、精度和舍入规则。

  • 记录汇率来源、源币种、目标币种、汇率值和汇率时间。

  • 同币种金额按币种精度计算和展示。

  • 跨币种场景生成换汇流水。

  • 换汇差额、手续费、汇损汇益单独展示。

  • 排行榜跨币种展示时固定展示币种和汇率时间。

  • 历史流水不因新汇率静默重算。

  • 这家 Shopee 店铺用什么结算币种。

  • 这个账户用什么账户币种。

  • 这个金额按什么币种精度计算和展示。

  • 发生跨币种时，使用哪一个汇率来源、哪一个汇率时间点、生成哪一笔换汇流水。

规则：

  • 店铺币种来自 Shopee 店铺所在站点或 Shopee 返回的结算币种。
  • 订单币种以 Shopee 订单和 escrow 明细为准。
  • 账户币种由账户创建时确定，创建后不修改。
  • 同币种结算不做汇率转换。
  • 跨币种提现、跨币种充值、跨币种补贴或平台内部折算，必须生成汇率记录和换汇流水。
  • 金额计算按币种精度配置处理，例如 TWD、USD、CNY 各自使用自己的精度规则。

14. 模块十一：提现、保证金与违约扣罚模块

14.1 模块使用者与系统行为

提现、保证金与违约扣罚模块处理用户提现、保证金门槛和管理员扣罚。用户发起提现后，系统先冻结金额，再根据第三方打款结果完成扣减或解冻。

使用者：

  • Shopkeeper：店主。发起佣金或保证金可提现部分提现。
  • Operator：运营。发起回款或保证金可提现部分提现。
  • Management：管理员。处理提现异常、执行违约扣罚。

系统行为清单：

  • 用户提交提现申请。
  • 系统读取账户类型和可用余额。
  • 保证金账户提现要校验最低保证金门槛。
  • 校验通过后冻结提现金额。
  • 提交第三方打款并记录手续费。
  • 打款成功后扣减冻结金额并生成流水。
  • 打款失败后解冻金额并标记失败。
  • 管理员手动提交违约扣罚原因并生成扣罚单。

提现规则：

  • 店主佣金余额可全额提现。
  • 运营回款余额可全额提现。
  • 平台不收提现手续费。
  • 第三方支付机构手续费由收款方承担。

保证金规则：

  • 店主保证金最低 15000 台币。
  • 店主保证金超出 15000 台币的部分可提现。
  • 运营保证金字段保留，当前最低门槛 0。
  • 店主违约时，管理员手动从店主保证金中扣除。
  • 运营违约时，管理员手动从运营保证金或运营回款中扣除。

  • 保证金扣罚必须管理员手动操作并填写原因。
  • 保证金低于门槛时，店主不能绑定新合作或继续新增可发货订单，具体拦截可由业务开关控制。

15. 模块十二：通知、任务与人工介入模块

15.1 模块使用者与系统行为

系统中很多节点不能只改状态，还要让人知道下一步做什么。

使用者：

  • SystemJob：系统任务。根据业务事件创建通知和待办。
  • Management：管理员。处理异常、超时升级、平台介入任务。
  • Shopkeeper：店主。处理授权失效、预付款不足、保证金不足等任务。
  • Operator：运营。处理发货、揽收失败、售后协作等任务。

系统行为清单：

  • 接收领域模块的通知请求。
  • 判断是否需要人工动作。
  • 不需要人工动作时只创建通知。
  • 需要人工动作时创建待办任务和通知。
  • 任务处理结果回写给原领域模块。
  • 任务超时后升级给管理员。
  • 人工处理必须记录审计。

需要通知的典型场景：

  • 店主 Shopee 授权失效。
  • 店主预付款不足。
  • 预付款不足超过 24 小时。
  • 订单已锁款，运营可发货。
  • 物流揽收失败，需要重新发货。
  • 订单进入取消或退货。
  • 账款调整失败，需要管理员重试。
  • 保证金低于最低门槛。

16. 模块十三：排行与经营看板模块

16.1 模块使用者与系统行为

业务补充要求后台首页提供排行榜，用于刺激运营和店主加入。

使用者：

  • Shopkeeper：店主。查看经营看板和排行榜。
  • Operator：运营。查看排行榜激励信息。
  • Management：管理员。查看后台首页运营数据。
  • SystemJob：系统任务。定时汇总已入账佣金流水生成快照。

系统行为清单：

  • 定时读取已入账佣金流水。
  • 按店铺和统计周期汇总佣金。
  • 过滤异常、冲销或未完成流水。
  • 生成每日或近 7 天排行榜快照。
  • 对店铺名称脱敏。
  • 使用占位图或打码图展示店铺 Logo。
  • 首页读取排行榜快照，不实时扫全量流水。

排行榜规则：

  • 以店铺为排名对象。
  • 按每日或近 7 天实际达成佣金排名。
  • 店铺 Logo 使用占位图或打码图。
  • 店铺名称脱敏展示，例如 天***符。
  • 盈利数据可展示。

“实际达成佣金”应使用已入账、可对账的账户流水，不使用未结算订单预计值。

18. 开发落地顺序

按以下顺序实现，避免先做页面导致领域规则落不住：

  1. 用户、角色、Shopee 授权、店铺绑定。
  2. 店铺合作关系、合约模板、AI 辅助审阅、管理员核对、签名/盖章签署版本。
  3. 账户初始化。
  4. 订单同步、订单历史、订单状态映射。
  5. 预付款充值、托管锁款账户、托管失败提醒。
  6. 运营发货任务、履约状态同步。
  7. escrow 明细同步、结算分账、账户流水。
  8. 退款退货、账款调整、已结算追加流水。
  9. 货币精度、汇率、换汇流水。
  10. 提现、保证金、违约扣罚。
  11. 通知任务、排行榜、经营看板。
